# PCB layout — placement pass (v2, 2026-07-21)

`clock.kicad_pcb`, generated by `gen/pcb_build.py` from `clock.kicad_sch`
(same generated-not-hand-edited convention as the schematic — edit the
script, not the `.kicad_pcb`, and re-run to regenerate). Built directly on
KiCad's `pcbnew` Python API, run under KiCad's own bundled interpreter:

```
/Applications/KiCad/KiCad.app/Contents/Frameworks/Python.framework/Versions/3.9/bin/python3.9 gen/pcb_build.py
```

**Scope: placement + stackup + zones + keepouts + full net assignment.
No traces are routed.** All 173 schematic parts are placed, sided, and
carry their net (visible as ratsnest in the KiCad GUI). Hand-routing ~200
nets across a 6-block, dual-rail, motor+RF+audio board isn't something
that converges to a *correct* result without a router or a human at a
screen — a first pass claiming to have routed it would likely be wrong in
ways nobody could see until fab. This gets the harder, judgment-heavy part
(what goes where, with what clearance, and why) done and verified; routing
is mechanical from here.

## v2 changes (this pass)

Rebuilt from the v1 layout in response to review: v1 was DRC-clean but
used a 0.6mm pad-to-pad gap almost everywhere, which is correct on paper
but not actually buildable — no hand-soldering room, no routing channel
between parts, connectors close enough to read as merged. Three real
fixes, in order of how much they changed:

1. **Schematic change: 5 of the 7 on-board NeoPixels moved off-board.**
   Only the two side pixels (dial-wash, 9 & 3 o'clock — renumbered
   D40/D41, chain positions 1–2) stay on the PCB. A new 3-pin JST-PH
   breakout, **J12** (`+5V`/`DATA`/`GND`), carries the chain out to 5 more
   SK6812 wired off-board in series (chain positions 3–7), so the status
   row can be placed around the dial by hand instead of competing for PCB
   edge space. Edited in `gen/b_led.py`, regenerated via `mksym.py` +
   `build.py` — ERC still returns just the 2 pre-existing known-benign
   items.
2. **Real spacing.** `grid_fill()`'s gap went from a flat 0.6mm to
   0.85–1.8mm depending on zone (denser only where a zone's items are
   genuinely tiny 0603s with no connectors), and connectors get 1.5x the
   local gap on top of that. Zones themselves now sit **≥2mm apart**
   (was 0mm — several v1 zones shared a boundary edge), which is also the
   routing channel between blocks and where the inner-layer GND pours get
   real width instead of being squeezed to a sliver between courtyards.
3. **microSD (J6) orientation, verified not assumed.** v1 guessed a
   rotation; this pass rendered the footprint's 3D model to actually see
   which edge the card slot opens from, then empirically tracked a
   contact pad vs. the socket's mouth-reinforcement tab through all 4
   rotations **on the flipped (back-side) footprint** (rotation interacts
   with the front→back flip, so "which way is the slot facing" isn't
   deducible from the unflipped footprint alone). Result: **rot=270°**
   puts the slot at +X. J6 sits at (100, 93), slot mouth ~1.7mm from the
   board's right edge, contacts facing inward toward its own SPI passives.

## Stackup

4-layer, 1.6mm. **In1.Cu and In2.Cu are both solid GND pours** (outline
placed; fill computes on first open in the GUI or `Edit → Fill Zones` —
`ZONE_FILLER.Fill()` segfaults under the bundled scripting interpreter, so
this is deferred rather than baked in). Two GND planes rather than a
single split power plane: six different rails (VBUS/BAT/+3V3/+5V/+12V/
PVDD) whose regions are already physically separated by the placement
grouping, so a hand-split power layer would mostly duplicate what
placement already gives, while risking two differently-named zones ending
up too close without a router in the loop to guarantee clearance.
GND-GND is also the better call for a board mixing an RF module, two
stepper drivers, and a class-D-ish audio amp — solid return path, easy
shielding between F.Cu and B.Cu.

Board minimum clearance set to **0.1mm (4mil)** rather than KiCad's 0.2mm
default — needed by J1's native 0.5mm-pitch USB-C pads and U5's SOT-563
pads, both per their own datasheets, not a placement error. 0.1mm is
within PCBWay's advanced/HDI capability, not their base tier — flag this
when ordering.

## Placement method

164 of 173 parts (all R/C/RT plus most ICs/discretes) are **grid-packed**
into 9 rectangular per-block zones (`FILL_ZONES` in `pcb_build.py`), not
hand-coordinated — the schematic's own block structure (`b_*.py`
generators) gives the grouping, and a real bin-packer (tallest-first
shelf packing, sized from each footprint's actual measured pad geometry,
gap tuned per zone) placed within a zone is more reliable than hand-picked
mm offsets at this component count. Only **9 parts have a genuine
hand-placed anchor**: M1 (motor, must be dead centre), U8 (ESP32, antenna
orientation), Y1 (crystal, proximity to U8), J6 (microSD, verified slot
orientation), and D40/D41 + their decoupling caps C230/C231 (front-side
LEDs pinned by the 9/3-o'clock constraint, with the caps nudged a few mm
off the exact LED position — see below).

Every zone rectangle is checked (`verify_zones()`) against 4 hard
exclusions before anything is placed: the ESP32 keepout, the motor's
courtyard (as a circle — see the centre-offset note below), the battery
footprint, and J6's footprint + slot mouth. This replaced an earlier
all-hand-coordinated ~54-anchor floor plan that looked reasonable on paper
and turned out to have 1000+ real overlaps once actually checked —
eyeballed mm-level spacing across 170+ parts isn't reliable without a live
renderer in the loop; a verified packer is.

### The motor's courtyard isn't centred on the shaft

Found via DRC, not derived up front: M1's footprint anchor — which is
also its courtyard circle's centre — sits 6mm *above* the shaft in local
coordinates (the shaft pad is at local (0,−6), and that offset survives
the flip-to-back unchanged, since flip mirrors about the anchor). M1 is
placed at (55, 61) precisely so the **shaft** lands at board-centre
(55, 55) — but the **body/courtyard** (radius ~16.7mm) is actually
centred at (55, 61). `MOTOR_COURTYARD_CENTER` in `pcb_build.py` uses the
correct point; every zone-safety distance check is against that, not the
shaft position. (First pass used (55, 55) for both, which put the
AUDIO_EXT zone 3 real components inside the true courtyard before this
was caught.)

### Front-side decoupling caps (C230/C231)

The schematic places one 100nF decoupling cap directly behind each
on-board pixel. (13, 55) and (97, 55) — directly behind D40/D41 — both
fall inside the POWER and MOTORDRV zone rectangles respectively; DRC
caught a real short risk between C231 and a MOTORDRV part sharing that
spot before this was fixed. Rather than reshaping those zones around 2
single points, C230/C231 sit a few mm off the exact LED position instead
(18, 48) and (103, 51), verified clear of every other placed footprint —
still a short via-hop from their pixel, still on the back (favouring
constraint 8), just not fighting the packer for the identical XY.

## Verification run

- `pcb_check.py` (custom, in `gen/`): pairwise courtyard-overlap, motor-
  keepout, and board-edge checks using each footprint's **pad geometry**,
  not `GetBoundingBox()` — the latter includes reference/value silkscreen
  text (some of it 50+ characters) and produced hundreds of false-positive
  "overlaps" in an earlier pass before this was caught. Final state: 0
  keepout violations, 0 board-edge overflows, 1 same-side "overlap" — D40
  (front) vs. F1 (back, a long-leaded TCO fuse whose bounding box spans
  its full lead length): confirmed benign by checking actual pad
  positions, 6mm+ apart — F1's *body* sits behind D40 on the opposite
  face, no physical or electrical conflict, same as any double-sided
  board with parts sharing an XY footprint on opposite faces.
- `kicad-cli pcb drc --severity-all`: **0 clearance violations, 0 real
  shorts from placement** (verified: the one non-obvious short DRC did
  catch, C231↔C185, was real and is fixed — see above). Remaining:
  - **~235 silk_overlap / silk_over_copper** — reference-designator text
    reflowing after packing; cosmetic, fixed in the GUI (Tools → Cleanup,
    or a `harvest.py`-style pass), zero effect on fabrication.
  - **35 courtyards_overlap** — mostly in the zones tuned to a tighter
    gap (1.0–1.3mm: WAKE, RAILS_IO, MOTORDRV) where courtyard margins
    (which extend beyond the pad-to-pad gap this layout actually
    controls) touch. 0 real clearance violations among them.
  - **13 shorting_items + 12 drill_out_of_range**, all on U8 pad "41" —
    the stock ESP32-S3-WROOM-1 footprint's exposed-pad/thermal-via array
    reports a duplicate pad number the schematic didn't net-map cleanly.
    Pre-existing footprint/schematic pairing quirk, not a placement
    artifact — needs a schematic-side fix (map the EP pin to GND
    explicitly), not a PCB one.
  - **2 pth_inside_courtyard** — down from 4-5 in earlier passes.

## Regenerating

```
cd gen
/Applications/KiCad/KiCad.app/Contents/Frameworks/Python.framework/Versions/3.9/bin/python3.9 pcb_build.py   # rebuild placement
/Applications/KiCad/KiCad.app/Contents/Frameworks/Python.framework/Versions/3.9/bin/python3.9 pcb_check.py   # overlap/keepout re-check
/Applications/KiCad/KiCad.app/Contents/MacOS/kicad-cli pcb drc --severity-all clock.kicad_pcb                 # full DRC
```

`pcb_build.py` re-derives the BOM straight from `clock.kicad_sch` every
run (never hand-transcribed), so schematic edits (new parts, footprint
swaps) flow through on the next build — you'd only need to touch
`ANCHORS` / `FILL_ZONES` if a *new* part needs a home. If you edit
`b_led.py` (or any block file) and regenerate the schematic, re-run
`pcb_build.py` afterward to pick up the change.
